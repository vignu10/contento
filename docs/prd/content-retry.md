# PRD: Content Retry Functionality

## Problem
When content processing fails (network error, API rate limit, etc.), users have no way to retry. They must delete and re-upload content, which is frustrating and wastes their time.

## Users
- Users whose content failed to process due to temporary issues
- Users who hit API rate limits during processing
- Users experiencing network glitches during upload/processing

## User Stories
- As a user, I want to retry processing failed content without re-uploading
- As a user, I want to know why my content failed so I can fix it
- As a user, I want to retry just the failed outputs (transcription or generation)

## Proposed Solution
Add retry functionality for failed content:

**Failed Content Badge**:
- Content cards with status "failed" show error badge
- Click on failed content to see details

**Content Detail Page**:
- If status is "failed", show error message
- "Retry" button to restart processing
- "Retry" replaces existing outputs (or appends, choose approach)

**Retry Logic**:
- Transcription retry: Re-run Whisper job
- Generation retry: Re-run AI generation job
- Both: Full pipeline re-run
- Keep original timestamp for "createdAt", update "processedAt"

**Error Messages**:
- Display specific error: "Network timeout", "API rate limit", "Invalid file format"
- Provide actionable feedback: "Please try again in a few minutes"
- Log errors for debugging

**Rate Limiting**:
- Limit retries to 3 per piece of content to prevent abuse
- Show retry count: "Retry 2 of 3"

## Scope
### In Scope
- Retry button on failed content detail page
- Retry transcription (if transcription failed)
- Retry generation (if generation failed)
- Display error messages for failures
- Limit retries to 3 per content
- Update status to "processing" → "completed" or "failed" on retry

### Out of Scope
- Automatic retry (without user action)
- Partial retry (retry only specific formats)
- Retry scheduling (retry in X minutes)
- Retry notifications

## Technical Approach
- **Frontend**: Add "Retry" button on content detail when status="failed"
- **API**: POST `/api/content/[id]/retry` endpoint
- **Backend**: Re-trigger queue jobs for transcription/generation
- **DB**: Reset status to "pending", increment retry count
- **Queue**: Use existing Bull queue infrastructure
- **Error handling**: Capture and display specific error reasons

## Acceptance Criteria
- [ ] Failed content shows error message
- [ ] "Retry" button appears for failed content
- [ ] Clicking retry updates status to "processing"
- [ ] Retry re-runs the appropriate job (transcription or generation)
- [ ] Retry succeeds where original attempt failed (if issue was transient)
- [ ] Retries are limited to 3 per content
- [ ] Error messages are clear and actionable
- [ ] Tests verify retry functionality

## Design Notes
- Use retry icon (refresh/rotate from lucide-react)
- Show "Retry 1/3" to indicate remaining attempts
- Disable retry button while processing
- If all retries fail, suggest deleting and re-uploading
